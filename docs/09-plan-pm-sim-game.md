# プランF: ITプロジェクト管理シミュレーションゲーム

## 概要

プレイヤーがITプロジェクトのPM（プロジェクトマネージャー）として、スプリント計画・リソース配分・リスク対応を行い、納期内・予算内でプロジェクトを成功させるシミュレーションゲーム。

**コンセプト**: 「ゲームで遊びながらPMの基礎が身につく」教育×娯楽（エデュテインメント）作品

> **なぜこのプランが面白いか**:  
> 自身のITエンジニア・PM経験が「ゲームの設計資産」になる。技術的にも（Unity C# / 状態管理 / データ駆動設計）深く掘れるうえ、**企業研修向けB2Bライセンス**という高単価な収益チャンネルが存在する。

## ゲームデザイン

### コアゲームループ

```
① プロジェクト受注
   顧客からの依頼（要件・予算・納期）を確認
        ↓
② スプリント計画（2週間×N回）
   チームメンバーにタスクを割り振る
   ・エンジニア: 実装スピード重視
   ・QA:        品質重視
   ・デザイナー: UX品質
        ↓
③ スプリント実行（時間経過）
   ランダムイベント発生
   ・バグ発覚 → 工数追加?
   ・クライアント仕様変更 → スコープ調整?
   ・メンバー体調不良 → リソース不足?
   ・技術的負債が爆発 → リファクタリング必要?
        ↓
④ スプリントレビュー
   進捗確認・バーンダウンチャート表示
   ステークホルダー満足度が変動
        ↓
⑤ 納品 or デスマーチ判定
   期限内完了 → 評価・報酬
   超過 → ペナルティ（ペット死亡ならぬチーム崩壊）
```

### 評価パラメータ（プレイヤーが管理するメーター）

| パラメータ | 説明 | 枯渇するとどうなるか |
|-----------|------|-------------------|
| **予算** | プロジェクト資金 | ゲームオーバー |
| **納期** | 残り日数 | 0日でクライアントが激怒 |
| **品質スコア** | バグの少なさ・UX品質 | 納品拒否・炎上 |
| **チームモラル** | メンバーの疲弊度 | 離職 → 戦力ダウン |
| **クライアント満足度** | 信頼関係 | 中途解約 |
| **技術的負債** | 手抜き実装の蓄積 | 突然バグ多発 |

### ランダムイベント例

| カテゴリ | イベント | 選択肢 |
|---------|---------|--------|
| 技術 | 「本番環境で重大バグ発覚」 | A: 即修正（工数-5日） B: リリース延期 |
| 人事 | 「キーパーソンが退職表明」 | A: 引き留め（予算-50万） B: 採用活動開始 |
| 要件 | 「クライアントが仕様追加要求」 | A: 受け入れ B: スコープ交渉 |
| 外部 | 「使用ライブラリがEOL」 | A: 即移行 B: 自己責任で継続 |
| チーム | 「エンジニアが過労気味」 | A: 休息指示 B: 突き進む |

### ゲームモード

| モード | 内容 |
|--------|------|
| **チュートリアル** | アジャイル入門（スプリント×3回の小プロジェクト） |
| **ウォーターフォール編** | 計画重視の古典的PM手法を体験 |
| **スタートアップ編** | 人材・予算が少ない中で最速でMVPを出す |
| **大規模プロジェクト編** | 50人体制・2年プロジェクトの複雑な調整 |
| **炎上案件編** | 引き継いだ炎上プロジェクトを立て直せるか |

## 技術スタック

### 推奨: Unity（既存スキル活用）

```
エンジン:       Unity 2022 LTS (2D)
言語:           C#
UI:             UI Toolkit (グラフ・ダッシュボード表示に最適)
データ管理:     ScriptableObject でイベント・スキルをデータ駆動化
状態管理:       Stateless / 独自 FSM (有限状態機械)
グラフ表示:     バーンダウンチャートは Unity Charts / Graphy
セーブ:         JSON シリアライズ (Newtonsoft.Json)
ローカライズ:   Unity Localization Package (日英対応)
配布:           Steam + itch.io + WebGL (ブラウザプレイ対応)
```

### アーキテクチャ設計

```
GameManager
├── ProjectState          # プロジェクト全体の状態
│   ├── Budget
│   ├── Deadline
│   ├── QualityScore
│   └── TechDebt
├── TeamManager           # チームメンバー管理
│   ├── Engineer[]
│   ├── QA[]
│   └── Designer[]
├── SprintSystem          # スプリント進行ロジック
│   ├── TaskQueue
│   ├── VelocityCalculator
│   └── BurndownTracker
├── EventSystem           # ランダムイベント
│   ├── EventPool (ScriptableObject)
│   └── EventResolver
└── UIController          # ダッシュボード表示
    ├── BurndownChart
    ├── ParameterPanel
    └── EventDialogUI
```

## 差別化ポイント

### 競合分析

| 競合 | 特徴 | 本作の優位性 |
|------|------|------------|
| Game Dev Tycoon | ゲーム会社経営 | IT開発現場に特化・より本格的 |
| Overcooked | チームワーク | PMの意思決定にフォーカス |
| 一般的な経営SIM | 抽象的な数値ゲーム | ITあるある・実務直結で共感性高 |
| 研修スライド | 受動的 | ゲームで能動的に体験・記憶に残る |

### 最大の差別化

**「実際のエンジニア・PMが作った」という信頼性**が最大の武器。

- バグ隠蔽イベント・デスマーチ・仕様変更地獄など **業界の「あるある」が詰まっている**
- Qiita / Zenn / X での発信で「エンジニアが作ったゲーム」として注目を集めやすい
- IT研修会社への営業時に「現場経験者が設計」は大きな説得力になる

## リスクと対策

| リスク | 対策 |
|--------|------|
| コンテンツ（イベント）が少なくて飽きる | 最初にイベント50種を設計してから実装開始 |
| ゲームバランス調整が大変 | ScriptableObject でデータ外部化 → Excelで調整 |
| B2B営業が難しい | まずB2Cで実績（レビュー・ダウンロード数）を積んでから営業 |
| 「教育ゲームは面白くない」イメージ | ストーリー・キャラクター・ユーモアで「まず面白い」を優先 |
