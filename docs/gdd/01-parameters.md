# GDD: パラメータ設計

## 基本方針

- 全パラメータは **0〜100** のスケールに統一（予算・日数は実数だが表示は%ゲージも併用）
- パラメータ同士は**相互に影響**し合う（モラル低下 → ベロシティ低下 → 進捗遅延）
- プレイヤーは毎スプリント末に「どのパラメータを犠牲にするか」を選ぶトレードオフを迫られる

---

## コアパラメータ（7つ）

### 1. 予算（Budget）
| 項目 | 値 |
|------|-----|
| 単位 | 万円 |
| 初期値 | 難易度による（300〜1000万円） |
| 表示 | 残額 + ゲージ（残額 / 初期予算） |
| ゲームオーバー条件 | 0円 |

**消費される場面**:
- 毎スプリント: チームの人件費（エンジニア1人 = 80万円/月）
- イベント選択肢A/B: 追加費用が発生するものが多い
- 残業・外注費用

**回復する場面**:
- 中間マイルストーン達成でクライアントから中間支払い
- 一部イベント選択肢（コスト削減策）

---

### 2. 残り日数（Deadline）
| 項目 | 値 |
|------|-----|
| 単位 | 日 |
| 初期値 | 難易度による（60〜180日） |
| 表示 | 残日数 + カレンダー形式 |
| ゲームオーバー条件 | 0日 かつ 進捗 < 完了条件 |

**減少する場面**:
- 毎ターン自動で経過
- 一部イベントで日数消費（「緊急会議3日間」等）

**増加する場面**:
- クライアント満足度が高い状態での「納期延長交渉」イベント成功時

---

### 3. 進捗（Progress）
| 項目 | 値 |
|------|-----|
| 単位 | % |
| 初期値 | 0% |
| 表示 | プログレスバー + スプリントバーンダウンチャート |
| 勝利条件の一つ | 100%到達（かつ品質スコア ≥ 40） |

**計算式**:
```
1スプリントの進捗増加 = チームベロシティ × (1 - 技術的負債補正) × モラル補正

チームベロシティ = Σ(各メンバーのスキル値) × 0.8  ※スプリント1回2週間
技術的負債補正   = 技術的負債 / 200              ※負債50なら25%減
モラル補正       = チームモラル / 100             ※モラル60なら60%効率
```

---

### 4. 品質スコア（Quality）
| 項目 | 値 |
|------|-----|
| 単位 | 0〜100点 |
| 初期値 | 80 |
| 表示 | グレード表示（S/A/B/C/D/F）+ 数値 |
| ゲームオーバー条件 | 納品時に 40 未満（クライアントが受け入れ拒否） |

**低下する場面**:
- テストをスキップした場合（スプリントでQAに割かない）
- 技術的負債が高い状態での開発
- 一部イベント選択肢（「速度優先」選択）

**回復する場面**:
- QAに工数を集中させたスプリント
- 「リファクタリングスプリント」の実施（進捗は増えない）
- 一部イベント選択肢（「品質改善」選択）

**グレード基準**:
| スコア | グレード | 効果 |
|--------|---------|------|
| 90〜100 | S | クライアント満足度+5/スプリント |
| 70〜89 | A | 変化なし |
| 50〜69 | B | クライアント満足度-2/スプリント |
| 30〜49 | C | クライアント満足度-5/スプリント + バグイベント率UP |
| 10〜29 | D | クライアント満足度-10/スプリント + バグイベント率大幅UP |
| 0〜9 | F | ゲームオーバー候補 |

---

### 5. チームモラル（Morale）
| 項目 | 値 |
|------|-----|
| 単位 | 0〜100 |
| 初期値 | 75 |
| 表示 | メンバーの顔アイコン（表情が変わる）+ 数値 |
| ゲームオーバー条件 | 0 かつ チームメンバーが全員離脱 |

**低下する場面**:
- 連続残業スプリント（1スプリント残業 → -10）
- 理不尽なイベント選択肢（「黙ってやれ」等）
- 進捗が計画より大幅遅延している状態が続く

**回復する場面**:
- 「チームランチ」イベント選択肢
- マイルストーン達成ボーナス（+15）
- 「リファクタリングスプリント」（達成感）

**モラルの閾値効果**:
| モラル | 状態 | ベロシティ補正 |
|--------|------|--------------|
| 80〜100 | 好調 | +20% |
| 60〜79 | 普通 | ±0% |
| 40〜59 | 不満 | -20% |
| 20〜39 | 危険 | -40% + 離職イベント率UP |
| 0〜19 | 崩壊寸前 | -60% + 強制離職イベント発生 |

---

### 6. クライアント満足度（Client Satisfaction）
| 項目 | 値 |
|------|-----|
| 単位 | 0〜100 |
| 初期値 | 70 |
| 表示 | クライアントキャラの表情 + 数値 |
| ゲームオーバー条件 | 0（契約解除） |

**低下する場面**:
- 進捗報告をサボる（報告スキップ）
- 品質スコアが低い状態が続く
- デモで重大バグが発覚
- 納期遅延の連絡をしない

**回復する場面**:
- 定期報告でポジティブな進捗を見せる
- 品質スコアが高い
- 先手を打った仕様変更の提案

**満足度の閾値効果**:
| 満足度 | 状態 | 効果 |
|--------|------|------|
| 80〜100 | 信頼 | 納期延長交渉が通りやすい |
| 60〜79 | 普通 | 変化なし |
| 40〜59 | 不安 | 仕様変更要求イベント率UP |
| 20〜39 | 不満 | 追加要件ラッシュイベント発生 |
| 0〜19 | 危機 | 契約解除ゲームオーバー |

---

### 7. 技術的負債（Tech Debt）
| 項目 | 値 |
|------|-----|
| 単位 | 0〜100 |
| 初期値 | 0 |
| 表示 | 「爆弾ゲージ」（赤くなると危険） |
| ゲームオーバー条件 | なし（ただし100に達すると大規模障害イベント強制発生） |

**増加する場面**:
- スプリントでQAをゼロにした場合（+15）
- イベントで「速度優先・コード品質後回し」を選択
- 「ハードコーディング」「コピペ実装」イベント選択

**低下する場面**:
- 「リファクタリングスプリント」の実施（進捗増加なし、負債-30）
- イベントで「コードレビュー強化」選択

**負債の影響**:
```
ベロシティへの影響  = 負債 / 200（負債100で50%減）
バグイベント発生率  = 基本率 + 負債 × 0.5%
品質スコア自然低下  = 負債 > 60 のとき毎スプリント -2
```

## 派生パラメータ（自動計算）

### チームベロシティ
```
ベロシティ = (チーム総スキル × モラル補正) × (1 - 技術的負債補正)

チーム総スキル  = Σ 各メンバーのスキル値（5〜20の範囲）
モラル補正     = max(0.4, チームモラル / 100)
負債補正       = min(0.5, 技術的負債 / 200)
```

### バグ発生率（イベント抽選用）
```
バグ発生率 = 基本5% + (100 - 品質スコア) × 0.3% + 技術的負債 × 0.5%
```

## 勝利・敗北条件

### 勝利
```
進捗 = 100%
かつ 品質スコア ≥ 40
かつ クライアント満足度 ≥ 1（契約中）
```

### 敗北（いずれか一つで即ゲームオーバー）
```
① 予算 = 0
② 残り日数 = 0 かつ 進捗 < 100%
③ クライアント満足度 = 0（契約解除）
④ チームメンバーが全員離脱
⑤ 技術的負債 = 100 かつ「大規模障害イベント」でBADENDを選択
```

### エンディングランク
| 条件 | ランク |
|------|--------|
| 進捗100% + 品質S + 満足度90+ + 納期内 | S（伝説のPM） |
| 進捗100% + 品質A以上 + 満足度70+ | A（優秀なPM） |
| 進捗100% + 品質B以上 + 満足度50+ | B（及第点） |
| 進捗100% + 品質C以下 or 満足度低 | C（なんとか完了） |
| 敗北条件に引っかかった | GAMEOVER |
