# DNSSEC導入ガイド〜DNS応答改ざん対策から実装まで

この記事では、DNSSEC（DNSSEC対応DNS）の導入方法と実装ステップを、セキュリティ実務の観点から解説します。特にインフラエンジニア・セキュリティエンジニアを対象に、ゼロトラスト環境でのDNS基盤強化の実践的な進め方をお伝えします。筆者は情報処理安全確保支援士（RISS）資格を保有し、Active DirectoryやSplunkを使った検知基盤の構築経験から、本ガイドを提供しています。

## DNSSECの概要と重要性

DNSSECは、DNS応答の完全性と真正性を保証する技術です。従来のDNSは平文で通信され、中間者攻撃やキャッシュポイズニング、DNS応答改ざんの脆弱性を抱えていました。

DNSSECは公開鍵暗号に基づき、DNS応答にデジタル署名を付与することで、以下を実現します：

- **真正性の確保**: DNS応答が本物の権威ネームサーバーから発行されたことを確認
- **完全性の保証**: DNS応答が転送中に改ざんされていないことを保証
- **チェーン検証**: ルートDNSSECキー→TLDキー→権威ネームサーバーキー の信頼チェーンを構築

ゼロトラスト環境では、「DNSも信頼せず検証する」という原則に基づき、DNSSECの導入は重要なセキュリティ施策です。

## DNSSEC非対応時の脆弱性シナリオ

DNSSECを導入せずDNSだけに頼った場合、以下のような攻撃シナリオが成立します。

### キャッシュポイズニング攻撃
再帰的リゾルバーに対し、攻撃者が偽の応答をより早く返す（または応答IDを予測する）ことで、権威ネームサーバーの応答よりも先に偽データをキャッシュさせます。結果、組織内のクライアントが攻撃者のIPアドレスにアクセスさせられます。

### DNS応答改ざん
ネットワーク上で傍受したDNS応答を改ざんし、別のIPアドレスを返すことで、ユーザーを偽のサイトへリダイレクトします。特に組織外からのVPN接続時、または境界検査が弱い環境で危険性が高まります。

### DNSSEC非対応環境での検知の難しさ
Splunkでシステムイベントログ（Windows Security Event 4688, 4689）やネットワークフローを監視していても、DNS応答レベルでの改ざんを検知するには、追加の検証が必要です。DNSSECがなければ、アプリケーションレイヤーでしか異常を検知できず、被害拡大を防ぐまでのリードタイムが長くなります。

## DNSSEC導入の実装ステップ

DNSSEC導入は、大きく3つのフェーズに分かれます。

### フェーズ1: 権威ネームサーバーへの署名配置

権威ネームサーバー（Bind9やNSD等）で、DNSSECキーペアを生成し、ゾーンデータに署名を付与します。

```bash
# Bind9を使用した例（※動作確認要）
cd /etc/bind/keys
dnssec-keygen -a RSASHA256 -b 2048 -n ZONE example.com
dnssec-signzone -e +365d -k Kexample.com.+008+<keyid>.key db.example.com Kexample.com.+008+<keyid>.key
```

生成されたRRSIG（署名）とDNSKEYレコードがゾーンファイルに追加されます。

### フェーズ2: 親ゾーン（レジストラ）でのDS登録

権威ネームサーバーのDSレコード（Delegation Signer）を親ゾーン（例：.comレジストラ）に登録します。これにより、クライアントが信頼チェーンを検証できるようになります。

```bash
# DS レコード情報の確認（※動作確認要）
dig example.com DS +short
```

レジストラ管理画面またはAPI経由で、生成されたDSレコードの値を登録します。

### フェーズ3: 再帰的リゾルバーでの検証有効化

組織内の再帰的DNSリゾルバー（Bind9, Unbound等）で、DNSSEC検証を有効化します。

```yaml
# Unboundの設定例（※動作確認要）
server:
  module-config: "validator iterator"
  auto-trust-anchor-file: "/etc/unbound/root.key"
  val-log-level: 2
```

クライアント（PC、サーバー、ネットワークデバイス）が、このリゾルバーを使用するよう設定します。

## Splunkでの DNS監視とDNSSEC検証ログ

DNSSEC導入後、Splunkダッシュボードで検証状態を監視することが重要です。

```splunk
index=dns sourcetype=dnsmasq OR sourcetype=unbound 
| search dnssec_status=* 
| stats count by dnssec_status, src_ip, query 
| where dnssec_status!="SECURE"
```

このクエリにより、DNSSEC検証に失敗したクエリ（BOGUS、INDETERMINATE等）を検出し、キャッシュポイズニングや改ざんの試みを早期に検知できます。RISS試験でも「ログ分析に基づいた脅威検知」が重要テーマであり、DNSSECログの監視体制はゼロトラスト実装の実践例となります。

## よくあるトラブルと対処法

### トラブル1: DNSSEC検証失敗（SERVFAIL応答）

**症状**: クライアントに SERVFAIL が返される  
**原因**: 権威ネームサーバーの署名が無効、またはDS登録が不完全  
**対処**:
- 権威サーバー側で `dnssec-verify` でゾーンを検証（※動作確認要）
- 親ゾーン側のDS登録が正しく反映されたか、`dig DS` で確認

### トラブル2: 署名有効期限切れ

**症状**: 日数経過後に突然 SERVFAIL が返される  
**原因**: RRSIG（署名）の有効期限切れ  
**対処**: 定期的な再署名スケジュール（通常365日ごと）を自動化し、cronで実行

```bash
# cronジョブ例（※動作確認要）
0 2 * * 0 /usr/local/bin/resign-zone.sh example.com
```

### トラブル3: キーローテーション時のダウンタイム

**症状**: キー更新時に一時的に検証失敗  
**原因**: 旧キーと新キーの共存期間不足  
**対処**: キーローテーション前に、事前に新キーをゾーンに追加し、十分な時間（TTL × 2倍以上）待機

## まとめ

DNSSEC導入により、DNS応答改ざん対策としてのゼロトラスト基盤が構築できます。本ガイドで解説した3フェーズ（権威サーバー署名、親ゾーン登録、リゾルバー検証）を順に進めることで、組織のDNS基盤をセキュアに強化できます。

要点：
- DNSSECは公開鍵暗号を使用し、DNS応答の完全性と真正性を保証する
- キャッシュポイズニングやDNS改ざん攻撃から組織を守る実践的な手段
- Splunkで検証状態を監視し、異常検知体制を整備することが重要
- 定期的な署名更新とキーローテーション管理は自動化すべき
